# Fix the Agent Launchpad

Goal: every step reflects **what the agent has actually done**, nothing false-positives, nothing sticks on "not started" after real completion. Once done, the launchpad collapses to a compact "you're set" card.

## Current bugs (verified against `useAgentLaunchpadProgress` + `AgentLaunchpad.tsx`)

1. **"Understand how matching works" resets on reload.** Completion is tied to local `matchingExpanded` state or `launchpad_completed_at` — but `launchpad_completed_at` only writes when the *whole* launchpad is finished, so this step effectively never persists on its own. Reloading marks it not-started again.
2. **"Review your matches" false-positives.** Marked done as soon as any `matches` row exists for the agent's exchange. Matches are auto-generated by `run-auto-matching` — the agent hasn't necessarily opened, approved, or acted on anything.
3. **"Move deals forward in Pipeline" false-positives.** Marked done when any `exchange_connections` row references an agent's exchange. Connections can appear before the agent takes any pipeline action.
4. **"Complete your profile" can look done when it's not (and vice-versa).** The check requires brokerage_name + brokerage_address + bio + ≥1 specialization all at once — a partially filled profile shows a hard "not started" instead of "in progress", and any of those four missing keeps the step red even if the agent thinks they're finished.
5. **Content is slightly stale.** Copy refresh pass for tone + accuracy (esp. "Create your first listing" → matches current wizard vocabulary; matching explainer wording).
6. **No compact completed state.** Once every step is done, the full checklist keeps rendering; the user asked for a "you're set" summary.

## Changes

### 1. Persist per-step acknowledgements (fixes bug #1, enables #2/#3)

Add three nullable timestamp columns to `public.profiles`:

- `launchpad_matching_ack_at` — set when the agent opens the "How matching works" explainer.
- `launchpad_matches_ack_at` — set on first visit to `/agent/matches` **or** first approve/send action.
- `launchpad_pipeline_ack_at` — set on first meaningful pipeline action (open a connection detail, or move a card).

RLS: covered by existing `profiles` self-update policy — no new policy needed.

### 2. Rewrite step completion in `useAgentLaunchpadProgress.ts`

| Step | New rule |
|---|---|
| profile | Unchanged rule, but expose partial state so the UI can show "in progress" (e.g. count filled fields). |
| client | ≥1 non-demo `agent_clients` row (unchanged, already workspace-scoped). |
| exchange | ≥1 non-demo `exchanges` row (unchanged). |
| matching | `profile.launchpad_matching_ack_at IS NOT NULL`. |
| matches | `profile.launchpad_matches_ack_at IS NOT NULL` **or** any match with `status IN ('approved','sent_to_client','interested')` on the agent's exchanges. |
| pipeline | `profile.launchpad_pipeline_ack_at IS NOT NULL` **or** any `exchange_connections` in status `accepted`/`in_progress`/`completed` on the agent's exchanges. |

Only Live workspace triggers acks (respect `isDemo`).

### 3. Wire the ack writes

- `AgentLaunchpad.tsx` → when the "matching" card expands, write `launchpad_matching_ack_at` (once) and invalidate the query.
- `AgentMatches.tsx` → on mount, if user is agent and ack is null, write `launchpad_matches_ack_at`.
- `AgentPipeline.tsx` (and/or `usePipelineStageOverride`) → on first stage change or connection open, write `launchpad_pipeline_ack_at`.

All writes are fire-and-forget (`.then()` ignore) and only run in Live mode.

### 4. UI polish in `AgentLaunchpad.tsx`

- Add an "in progress" status chip to `LaunchpadChecklistCard` (`LaunchpadStepStatus = 'done' | 'in_progress' | 'attention' | 'todo'`) — used by profile step when 1–3 of 4 fields are filled.
- Copy refresh across `src/content/agentLaunchpad.ts`:
  - Align "Create your first listing" with wizard's current vocabulary (property + criteria + financials).
  - Tighten matching explainer text.
  - Small tone pass for tips.
- **Compact completed state:** when `allStepsComplete`, render only the green summary card ("You're all set!") + a collapsible "Show checklist" toggle. The full grouped list stays hidden by default. `handleCompleteLaunchpad` still persists `launchpad_completed_at` so the nav item can hide/collapse if desired later.

### 5. Tests

- Extend `agent-launchpad-progress.test.ts`:
  - matching step true iff `launchpad_matching_ack_at` set.
  - matches step true via ack OR approved-status match, false with only auto-generated `new` matches.
  - pipeline step true via ack OR accepted+ connection, false with only pending.
  - profile "in_progress" derivation.
- Update `agent-launchpad-content.test.ts` for any copy-order assertions (order stays the same, so likely no change).

## Technical notes

- Migration: `ALTER TABLE public.profiles ADD COLUMN launchpad_matching_ack_at timestamptz`, same for `_matches_ack_at`, `_pipeline_ack_at`. No new grants (profiles already granted). No RLS change.
- Query: extend `select` in `fetchAgentLaunchpadProgress` to pull the three new columns; recompute `completionMap` from them.
- Ack writers use `supabase.from('profiles').update({...}).eq('id', uid).is('launchpad_<x>_ack_at', null)` so they're idempotent and cheap.
- No changes to `run-auto-matching`, RLS, or edge functions.

## Out of scope

- Adding/removing/reordering launchpad steps.
- Nav visibility rules once complete (keeping current behavior; only the page body compacts).
- Any dashboard or pipeline redesign.
